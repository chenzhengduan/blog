## 2. 错误查询和筛选演示

### 2.1 基础筛选操作

**演示步骤：**
1. **环境筛选**
   ```
   点击环境下拉框 → 选择 "production"
   或在搜索框输入：environment:production
   ```

2. **时间范围筛选**
   ```
   点击时间选择器 → 选择 "Last 24 hours" 或自定义范围
   ```

3. **错误状态筛选**
   ```
   点击状态标签：Unresolved（未解决）/ Resolved（已解决）
   ```

### 2.2 高级查询演示

**实用查询示例演示：**

#### 查询一：支付模块错误
```
搜索框输入：tag:module:"payment" AND environment:production

演示说明：
- 展示筛选结果
- 解释查询语法
- 说明标签的重要性
```

#### 查询二：特定用户错误
```
搜索框输入：user.id:"12345" OR user.email:"*@example.com"

演示说明：
- 如何追踪特定用户的问题
- 通配符的使用方法
- 用户上下文的价值
```

#### 查询三：移动端兼容性问题
```
搜索框输入：tag:platform:"mobile" AND browser.name:"Mobile Safari"

演示说明：
- 设备和浏览器信息的筛选
- 兼容性问题的快速定位
- 多条件组合查询
```

#### 查询四：新版本问题监控
```
搜索框输入：release:"my-app@2.0.0" AND first_seen:>2024-01-15

演示说明：
- 版本发布后的问题监控
- 时间范围的精确控制
- 新问题的快速识别
```

### 2.3 保存和分享查询

**演示操作：**
1. 创建复杂查询后，点击 "Save Search"
2. 命名查询（如："支付模块生产环境错误"）
3. 设置为团队共享
4. 演示如何快速访问保存的查询

## 4. 性能监控演示

### 4.1 页面性能指标

**截图要点：**
- Web Vitals 指标图表
- 页面加载时间分布
- 不同页面的性能对比
- 设备和网络环境影响

**演示说明：**
```
Web Vitals 指标解读：
├── LCP (Largest Contentful Paint)
│   ├── 好：< 2.5s
│   ├── 需要改进：2.5s - 4.0s
│   └── 差：> 4.0s
├── FID (First Input Delay)
│   ├── 好：< 100ms
│   ├── 需要改进：100ms - 300ms
│   └── 差：> 300ms
└── CLS (Cumulative Layout Shift)
    ├── 好：< 0.1
    ├── 需要改进：0.1 - 0.25
    └── 差：> 0.25
```

### 4.2 接口性能分析

**演示步骤：**
1. **响应时间分析**
   ```
   查看 P50、P75、P95、P99 响应时间
   识别性能异常的接口
   分析响应时间趋势
   ```

2. **错误率统计**
   ```
   按状态码查看接口错误分布
   识别高错误率的接口
   分析错误率变化趋势
   ```

3. **吞吐量分析**
   ```
   查看接口调用量变化
   识别高并发时段
   分析性能瓶颈
   ```

## 5. 告警配置演示

### 5.1 创建告警规则

**演示步骤：**
1. **进入告警配置**
   ```
   导航：Alerts → Alert Rules → Create Alert Rule
   ```

2. **配置告警条件**
   ```
   示例：错误率告警
   - 条件：error rate > 5%
   - 时间窗口：5 minutes
   - 环境：production
   ```

3. **设置通知方式**
   ```
   - 邮件：team@example.com
   - Slack：#alerts 频道
   - 钉钉：开发团队群
   ```

### 5.2 告警规则类型演示

**实用告警规则示例：**

#### 规则一：新错误告警
```
名称：新错误立即通知
条件：first_seen event
筛选：environment:production
通知：立即发送到 Slack
```

#### 规则二：错误率异常
```
名称：生产环境错误率异常
条件：error rate > 3% in 5 minutes
筛选：environment:production
通知：邮件 + 钉钉
```

#### 规则三：用户影响告警
```
名称：大量用户受影响
条件：affected users > 100 in 10 minutes
筛选：environment:production
通知：电话 + 邮件（高优先级）
```

## 6. 实际案例演示脚本

### 案例一：支付页面错误排查

**演示脚本：**
```
1. 背景介绍
   "假设我们收到用户反馈，支付页面出现异常，让我们用 Sentry 来排查"

2. 快速定位
   输入查询：tag:module:"payment" AND environment:production
   "可以看到支付相关的错误列表"

3. 选择错误
   点击错误率最高的问题
   "这个错误影响了 50+ 用户，我们来详细分析"

4. 堆栈分析
   查看错误堆栈和代码上下文
   "错误发生在支付表单验证函数中"

5. 用户上下文
   查看受影响用户的设备信息
   "主要影响 iOS Safari 用户"

6. 操作轨迹
   查看用户操作面包屑
   "用户在选择支付方式后触发错误"

7. 解决方案
   "根据分析，这是一个 iOS Safari 兼容性问题，需要修复表单验证逻辑"
```

### 案例二：性能问题分析

**演示脚本：**
```
1. 性能概览
   "我们注意到页面加载时间有所增加，来看看性能数据"

2. 指标分析
   查看 LCP 指标变化
   "LCP 从 2.1s 增加到 3.5s，超过了良好标准"

3. 页面对比
   对比不同页面的性能
   "首页和商品详情页性能下降最明显"

4. 设备分析
   按设备类型查看性能
   "移动设备受影响更严重"

5. 优化建议
   "需要优化图片加载和减少 JavaScript 包大小"
```

### 案例三：新版本监控

**演示脚本：**
```
1. 版本发布监控
   "我们刚发布了 v2.0.0，来监控新版本的表现"

2. 版本对比
   使用查询：release:"my-app@2.0.0"
   "新版本的错误率比上个版本高了 20%"

3. 新错误识别
   筛选 first_seen:>2024-01-15
   "发现了 3 个新的错误类型"

4. 影响评估
   查看受影响用户数和错误频率
   "影响了 200+ 用户，需要紧急处理"

5. 决策建议
   "建议立即发布热修复版本或考虑回滚"
```

## 7. 演示技巧和注意事项

### 7.1 演示技巧

1. **准备充分的数据**
   - 确保有真实的错误数据
   - 准备不同类型的错误案例
   - 设置好用户上下文信息

2. **流畅的操作**
   - 提前熟悉界面操作
   - 准备好查询语句
   - 标记重要的截图位置

3. **清晰的解释**
   - 每个操作都要解释目的
   - 强调关键信息的价值
   - 联系实际业务场景

### 7.2 常见问题处理

1. **数据不够丰富**
   - 提前在测试环境制造一些错误
   - 使用 Sentry 的演示数据
   - 准备截图作为备选方案

2. **网络问题**
   - 准备离线截图
   - 提前下载重要页面
   - 准备备用演示环境

3. **权限问题**
   - 确认演示账号权限
   - 准备管理员账号备用
   - 提前测试所有功能

### 7.3 互动环节设计

1. **让观众参与查询**
   ```
   "大家觉得应该用什么查询条件来找到移动端的错误？"
   "如果要查找昨天新出现的错误，应该怎么写查询？"
   ```

2. **案例分析讨论**
   ```
   "看到这个错误信息，大家觉得可能是什么原因？"
   "根据用户操作轨迹，我们能得出什么结论？"
   ```

3. **最佳实践分享**
   ```
   "在你们的项目中，最希望监控哪些类型的错误？"
   "大家觉得告警频率设置多少比较合适？"
   ```
