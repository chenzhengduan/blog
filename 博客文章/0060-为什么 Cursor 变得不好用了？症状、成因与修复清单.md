# 为什么 Cursor 变得不好用了？症状、成因与修复清单

> 一段时间以来，许多开发者反馈 Cursor 的体验“变差了”：理解力下降、编辑跑偏、延迟变高、上下文丢失等。本文给出系统性的观察维度、常见根因、快速修复方案与长期优化策略，帮助你把可控因素全部调优到位。

## 0. 结论先行（TL;DR）
- **先排查可控项**：模型/路由、网络/代理、扩展冲突、索引/缓存、项目体量与上下文边界。
- **快速止血**：固定模型与路由、重建索引、精简扩展、降低对话粘滞度（短回合+文件指引）、使用白名单上下文。
- **长期优化**：模板化提示工程、分层任务流、自动化测试兜底、采样对比与效果回归。

---

## 1. 你可能遇到的“变差”症状
- **上下文丢失**：重复问、记不住约定；对已有代码风格不对齐。
- **编辑跑偏**：越权大改、跨文件“顺手”改动、未遵守最小变更原则。
- **幻觉与瞎补**：编造 API/类型/文件；对第三方库接口猜错。
- **延迟与卡顿**：生成慢、Diff 卡住、索引长时间转圈。
- **规则被忽略**：不执行“先评审后修改”、不遵循项目代码规范、Commit 信息风格混乱。
- **MCP/联网不稳定**：检索不到文件、外部搜索超时、工具调用失败。

---

## 2. 常见根因总览（按可控程度排序）
1. **模型/路由波动**：模型版本更替、Provider 限流/熔断、走了降级路径。
2. **网络与代理**：丢包、被限速、DNS 污染、跨境链路不稳导致输出断流。
3. **扩展冲突或过多**：多 AI 插件并存（Copilot、Continue、Trae 等）抢占快捷键/上下文。
4. **项目体量超限**：索引不完整；多仓/单仓超大导致召回低、检索误召。
5. **索引/缓存损坏**：旧向量残留、工程改名/大搬迁后未全量重建。
6. **设置不当**：自动应用大范围 edits、过于宽松的上下文采样、启用危险格式化。
7. **提示工程缺失**：无模板、长对话漂移、缺少测试约束与验收条件。

---

## 3. 快速修复（20 分钟内可完成）
- **固定模型与路由**
  - 在设置中显式选择稳定模型（如同一供应商的指定版本），避免“自动最佳”。
  - 若支持，关闭“智能路由/自动降级”。
- **重建项目索引**
  - 删除本地索引缓存（Cursor 的工作区缓存与 `.cursor`/`.cache` 类目录），触发全量重索引。
  - 大仓请添加 `ignore` 规则，排除 `dist`、`node_modules`、大体积数据集。
- **精简与隔离扩展**
  - 暂时禁用除 Cursor 外的所有 AI/格式化/重构类扩展，验证冲突。
  - 逐个恢复，定位问题插件。
- **约束上下文与修改范围**
  - 采用“文件白名单”提示：明确只能编辑的路径与函数名。
  - 要求“先方案后实现”，并写入提示模板。
- **短回合交互**
  - 将复杂需求拆为 5-15 分钟可完成的子任务；每回合附上验收条件与测试要点。
- **网络与代理**
  - 切换稳定代理出口与 DNS；对海外 API 走固定高质量线路。

> 小贴士：出现“突然变傻”的错觉时，先换模型/路由+重建索引，80% 情况立刻改善。

## 4. 配置与模板建议

### 4.1 对话模板（粘贴到每次新会话首条）
```text
角色：资深工程助理。要求：
- 严格执行“先方案评审→再最小改动实现→新增测试→提交说明”。
- 只在白名单文件/函数内修改，禁止“顺手”跨文件。
- 若信息不足，必须列出缺口并停下等待确认。
- 输出必须包含：修改点列表、风险与回滚、测试要点。
白名单：
- 路径：app/components/*, src/services/user.ts
- 函数：updateUserProfile, fetchUser
禁止：全局重构、包升级、风格化无关改动。
```

### 4.2 PR/Commit 模板
```text
feat(scope): 简述本次变更

变更：
- [ ] 文件A：函数B：原因/影响
- [ ] 文件C：类型D：原因/影响

测试：
- [ ] 单元/集成用例编号
- [ ] 验证步骤与期望结果

风险与回滚：
- 风险点1/回滚策略
```

### 4.3 忽略与索引优化
- 在工作区忽略以下路径：`node_modules`、`dist`、`build`、二进制与数据集目录。
- 采用“多仓拆分 + 局部打开工作区”的方式，降低索引噪声。

---

## 5. 长期优化：把效果稳定在“可预期”
- **分层任务流**：需求评审（自然语义）→ 技术方案（结构化）→ 代码变更（最小集）→ 测试/回归（自动化）。
- **Few-shot 库**：沉淀 5-10 个高质量 Q→A/方案→实现→测试范例，作为上下文锚点。
- **观测与回归**：每周抽样 5 个典型任务做 A/B（模型/设置）对比；用脚本统计“被拒绝 edits 比例”“回滚率”。
- **CI 安全阈**：引入 Lint/Type/Tests 的硬门槛，阻断有风险的自动化改动。
- **工具编排**：让 Cursor 专注“编辑与重构”，搜索/抓取交由 MCP 或外部脚本完成，避免幻觉。

---

## 6. 故障排查清单（Checklist）
- [ ] 明确：问题是“变慢/变傻/跑偏/失忆”中的哪一类？
- [ ] 切换并固定模型与路由；记录版本号与时间窗口。
- [ ] 清缓存/重索引；确认忽略规则生效。
- [ ] 禁用除 Cursor 外的扩展；逐个恢复定位冲突。
- [ ] 将任务拆小，应用对话与提交模板；白名单约束修改范围。
- [ ] 走稳定网络与代理；切换 DNS。
- [ ] 建立 few-shot 库与每周回归机制。

---

## 7. 结语
工具的“智能波动”难以完全避免，但工程化的约束与流程可以把波动压缩在可控区间。与其“感觉变差”，不如把问题拆解为可观测的指标，用模板、索引与流程把稳定性夺回来。
