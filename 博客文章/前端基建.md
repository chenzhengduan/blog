# 前端基建文档目录

## 1. 项目概览

### 1.1 背景与目标

#### 项目背景

随着业务的快速发展，我们的前端应用规模不断扩大，开发团队也日益壮大。在这一过程中，我们面临以下挑战：

- **技术栈迁移**：从旧的技术栈向 Vue3 生态系统迁移
- **开发效率**：随着项目规模增大，开发效率面临挑战
- **协作成本**：多团队协作开发导致的沟通和协作成本增加
- **代码质量**：缺乏统一的规范和标准导致代码质量参差不齐
- **维护成本**：项目维护成本不断增加

#### 项目目标

基于以上背景，我们启动了前端基建项目，旨在：

1. **标准化**：建立统一的开发规范和标准，提升代码质量
2. **模块化**：通过微前端架构实现业务模块的解耦和独立部署
3. **自动化**：构建自动化的开发、测试、部署流程，提升开发效率
4. **组件化**：建立统一的组件库和工具库，提高代码复用率
5. **可观测性**：实现全链路监控和性能分析，快速定位并解决问题

### 1.2 技术栈说明（Vue3 + micro-app + Vite/Webpack）

本项目采用如下技术栈进行开发：

#### 前端框架

- **Vue 3**：核心前端框架，采用 Composition API 开发模式
- **TypeScript**：提供类型检查和代码智能提示，增强代码健壮性
- **Pinia**： Vue 3 推荐的状态管理库，替代 Vuex
- **Vue Router 4**：路由管理

#### 微前端解决方案

- **micro-app**：基于 Web Components 的微前端框架
  - 低侵入性：子应用几乎不需要改造
  - 简单直观：类似 HTML 标签的使用方式
  - 完善的沙箱机制：JS 沙箱、样式隔离、元素隔离

#### 构建工具

- **Vite**：主应用和新子应用采用 Vite 构建，提供更快的开发体验
- **Webpack**：兼容旧子应用的 Webpack 构建配置

#### 其他工具

- **ESLint + Prettier**：代码规范和格式化
- **Husky + lint-staged**：代码提交前检查
- **Commitizen + CommitLint**：规范化提交信息
- **Jest**：单元测试框架
- **Cypress**：端到端测试框架

## 3. 项目脚手架规范

### 3.1 脚手架功能设计

我们的项目脚手架基于 Vite 和 Vue CLI 进行定制开发，旨在提供一站式的项目初始化和开发体验。

#### 脚手架核心功能

1. **项目模板创建**
   - 主应用模板（Vue3 + TS + Vite）
   - 子应用模板（Vue3 + TS + Vite）
   - 兼容模板（Vue2 + Webpack）
   - 组件库模板

2. **交互式项目配置**
   - 项目名称、描述、作者等基本信息
   - 可选功能（TypeScript、ESLint、Prettier、Jest、Cypress等）
   - 样式方案（SCSS/LESS/Tailwind CSS）
   - 状态管理（Pinia/Vuex）

3. **开发工具集成**
   - 代码规范工具（ESLint + Prettier）
   - Git Hook工具（Husky + lint-staged）
   - 提交信息规范（Commitizen + CommitLint）
   - 单元测试框架（Jest）
   - 端到端测试框架（Cypress）

4. **自动化集成**
   - GitHub Actions 工作流配置
   - GitLab CI/CD 配置
   - 持续集成报告生成

### 3.2 项目初始化流程

我们的项目初始化流程采用交互式的方式，简化开发者的操作步骤。

#### 初始化流程图

```
开发者输入初始化命令
       ↓
选择项目类型（主应用/子应用/组件库）
       ↓
输入项目基本信息（名称、描述等）
       ↓
选择技术特性（TS、样式方案、状态管理等）
       ↓
选择开发工具（ESLint、Prettier、测试框架等）
       ↓
选择CI/CD配置（GitHub Actions/GitLab CI）
       ↓
生成项目文件结构
       ↓
安装依赖并初始化Git仓库
       ↓
初始化完成，显示下一步操作指引
```

#### 初始化命令

```bash
# 全局安装脚手架
$ npm install -g @company/frontend-cli

# 创建新项目
$ frontend-cli create my-project

# 使用特定模板创建项目
$ frontend-cli create my-project --template main-app
```

### 3.3 常用命令与 NPM Script 约定

为了统一开发体验和提高开发效率，我们对脚手架生成的项目中的 NPM Script 进行了标准化约定。

#### 脚手架命令

| 命令 | 描述 |
|---------|------|
| `frontend-cli create <name>` | 创建新项目 |
| `frontend-cli add <plugin>` | 添加插件到现有项目 |
| `frontend-cli update` | 更新项目配置和依赖 |
| `frontend-cli info` | 显示环境信息 |
| `frontend-cli lint` | 运行代码规范检查 |
| `frontend-cli test` | 运行测试 |
| `frontend-cli build` | 构建项目 |
| `frontend-cli deploy` | 部署项目 |
| `frontend-cli generate <type>` | 生成组件、页面等模板代码 |

#### 标准 NPM Script

所有项目的 `package.json` 中应包含以下标准化的脚本命令：

```json
{
  "scripts": {
    "dev": "启动开发服务器",
    "build": "构建生产版本",
    "build:test": "构建测试环境版本",
    "build:staging": "构建预发环境版本",
    "preview": "本地预览生产构建",
    "lint": "运行 ESLint 检查",
    "lint:fix": "运行 ESLint 检查并自动修复",
    "lint:style": "运行样式检查",
    "test": "运行所有测试",
    "test:unit": "运行单元测试",
    "test:e2e": "运行端到端测试",
    "commit": "使用 commitizen 提交变更",
    "prepare": "安装 husky hooks",
    "release": "发布新版本",
    "analyze": "分析构建产物大小"
  }
}
```

#### 环境变量约定

我们使用标准的环境变量来控制不同环境的配置：

| 环境变量 | 描述 | 默认值 |
|--------------|------|--------|
| `NODE_ENV` | 运行环境 | `development`/`production`/`test` |
| `VITE_APP_ENV` | 应用环境 | `development`/`test`/`staging`/`production` |
| `VITE_APP_API_BASE_URL` | API基础路径 | 根据环境不同设置 |
| `VITE_APP_PUBLIC_PATH` | 公共资源路径 | `/` |
| `VITE_APP_ENABLE_MOCK` | 是否启用Mock数据 | `false` |

### 3.4 脚手架扩展机制

我们的脚手架采用插件化设计，支持通过插件扩展功能。

#### 插件系统

脚手架插件系统允许开发者扩展脚手架的功能，主要包括以下类型的插件：

1. **模板插件**：添加新的项目模板
2. **功能插件**：添加新的功能（如图表库、国际化等）
3. **命令插件**：添加新的命令行命令
4. **构建插件**：自定义构建过程

#### 插件开发流程

1. **创建插件**：使用脚手架提供的插件模板创建新插件

   ```bash
   $ frontend-cli create-plugin my-plugin
   ```

2. **开发插件**：按照插件 API 开发功能

   ```javascript
   // 插件入口文件 index.js
   module.exports = (api, options) => {
     // 注册命令
     api.registerCommand('my-command', {
       description: '我的自定义命令',
       usage: 'frontend-cli my-command [options]',
       options: {
         '--option': '命令选项描述'
       }
     }, (args) => {
       // 命令实现
     })
     
     // 扩展生成器
     api.extendGenerator('component', {
       prompts: [ /* 自定义提示 */ ],
       actions: [ /* 自定义动作 */ ]
     })
     
     // 添加钩子
     api.afterProjectGenerate(context => {
       // 项目生成后的操作
     })
   }
   ```

3. **发布插件**：将插件发布到公司私有 NPM 或公共 NPM

   ```bash
   $ npm publish
   ```

4. **使用插件**：在项目中安装并使用插件

   ```bash
   $ frontend-cli add my-plugin
   ```

#### 插件开发规范

1. **命名规范**：插件包名应以 `frontend-cli-plugin-` 开头
2. **文档规范**：插件应提供完善的 README 文档
3. **版本管理**：遵循语义化版本控制规范
4. **测试要求**：插件应包含单元测试
5. **兼容性**：指定插件兼容的脚手架版本范围
