# 预检查增量校验测试流程

## 1. 场景说明
- 预检查开关开启后应保持开启状态（非手动不关闭）。
- 首次仅对“可视区 + 缓冲”范围内的行发起预检查。
- 滚动时按可视区增量触发新的预检查请求；已检查/已挂起的行不重复请求。
- 预检查列仅在`preCheckedIds`内的行显示状态；不在集合内不显示。
- 已有结果的行发生改动后：保存成功才触发该行的单行复检；保存失败不触发。

## 2. 手工测试流程（页面操作）

### 2.1 登录与进入排课
1) 打开登录：`http://localhost:3011/api/login/login?name=admin@test&pwd=xiaogj.com`
2) 打开排课：`http://localhost:3011/wtwo#/scheduleManage`
3) 切换到“表格排课”页签，等待表格数据加载完成。
4) 看到数据后

### 2.2 首次开启预检查
1) 开启“预检查冲突与限制”。
2) 在弹框中勾选“确认已经整理完成”并点击“开始检查”。（未勾选将无法开始，请务必先勾选再点击）
3) 期望：
   - 网络面板出现首次检查请求，ID 数量≈“可视区+缓冲”行数，明显小于全量。
   - 预检查开关保持开启，不会自动关闭。
4) 重要提醒：点击“开始检查”后需继续进行“2.3 滚动增量检查”以验证增量触发是否生效。

### 2.3 滚动增量检查
1) 缓慢向下滚动表格（每次 1 屏，停顿 300–500ms）。
2) 期望：
   - 持续出现新的检查请求，仅包含“新进入可视区+缓冲”的未检查/未挂起行ID。
   - 已检查过的行滚回视区内不重复请求。
   - 预检查列只在参与检查的行显示状态；未参与的不显示。

### 2.4 修改后单行复检（联调项）
1) 在已显示预检查结果的行，修改任一字段并保存。
2) 期望：
   - 保存成功后发起该行的单行检查请求，返回后更新该行的预检查状态显示。
   - 保存失败不触发自动检查。

### 2.5 手动关闭
1) 手动关闭“预检查冲突与限制”。
2) 期望：
   - 预检查列隐藏；滚动不再触发检查请求。

## 3. Playwright 自测脚本步骤

> 说明：以下为步骤脚本化思路，便于后续实现自动化。

### 3.1 初始化
- 打开登录页，等待成功；跳转排课页。
- 点击“表格排课”，等待表格主体出现（选择器可为表头文字或列表容器类名）。

### 3.2 首次检查断言
- 点击开关，勾选确认并“开始检查”。
- 断言：
  - 出现检查请求；请求体 ID 数量 > 0 且小于总行数。
  - 页面上开关处于“开启”状态。

### 3.3 滚动增量断言
- 对表格滚动容器多次执行 `scrollTop += 700`，每次等待 400ms。
- 断言：
  - 新的检查请求持续出现；各次请求的 ID 集不重复（或与已检查集合取差集后仍 > 0）。

### 3.4 单行复检断言（联调）
- 编辑一行已检查过的数据并保存。
- 断言：
  - 保存成功后出现仅包含该行 ID 的检查请求；该行预检查列状态更新。

## 4. 观察点与问题定位
- 若滚动后无新请求：检查 `scrolling` 回调是否触发、防抖是否生效、`calcCurrentRangeWithBuffer` 范围计算是否合理、`checkedIds/pendingIds` 去重是否误判。
- 若开关被动关闭：检查开启流程中的 Promise `catch` 分支是否改动开关状态。
- 若列显示异常：确认 `preCheckedIds` 的维护与渲染条件一致。

## 5. 后续优化建议
- 将已检查失败的 ID 放入 `retryQueue`，在下次滚动或按钮触发时重试。
- 对非常长页面可调大缓冲区或调小防抖间隔，权衡性能与时效性。
