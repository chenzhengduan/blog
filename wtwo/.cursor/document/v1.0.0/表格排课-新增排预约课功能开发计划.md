### **开发计划文档：表格排课 - 新增“排预约课”类型**

---

#### **1. 需求背景与目标**

当前表格排课功能支持“给班级排课”和“给学员排课”两种模式。本次需求旨在引入第三种模式——**“排预约课”**。此模式的核心特点是不直接关联到特定的班级或学员，而是创建一个可供学员后续预约的课程模板。

为支持此功能，需要在表格排课界面 (`class-table-course.vue`) 中进行一系列改造，使其能够根据选择的排课类型 (`selectedTableType`) 动态调整表格的列结构、数据处理及校验逻辑。

#### **2. 需求边界与关键点分析**

**“排预约课”模式的核心特征：**

*   **无班级/学员列**：不显示“上课班级”和“上课学员”列。
*   **新增专属列**：
    *   **可约人数 (`MaxStudentCount`)**: `number` 类型，非必填，允许的最小值是`0`。
    *   **开课人数 (`StartStudentCount`)**: `number` 类型，非必填，允许的最小值是`0`。
*   **字段联动逻辑**：
    *   **课程类型影响**：当选择的“上课课程”为“一对一”类型时，`可约人数`和`开课人数`自动填充为 `1` 且变为**不可编辑**状态。选择其他课程或清除课程时，恢复为可编辑状态并清空数值。
    *   **人数校验**：`开课人数`的值不能大于`可约人数`的值（如果两者都填写了）。
*   **默认值**：
    *   新增一行时，“开放预约”(`IsSubscribeCourse`) 字段默认为 `1` (是)。

**与现有逻辑的关联与冲突分析：**

| 功能点 | 班级排课 (10) / 学员排课 (20) | 排预约课 (30) | 冲突与解决方案 |
| :--- | :--- | :--- | :--- |
| **核心关联列** | 有 `ClassID` 或 `StudentUserID` | 无 `ClassID` 和 `StudentUserID` | **无直接冲突**。通过 `getCurrentTableColumns` 动态加载不同列配置即可解决。 |
| **分组功能** | 可按班级/学员分组 | 不能按班级/学员分组 | **存在冲突**。在切换到“排预约课”时，如果当前分组条件是“班级”或“学员”，该分组需要被自动清除。**（此问题在上次修改中已解决）** |
| **数据初始化** | `useTableData.ts` 中的 `addEmptyRow` 未包含新字段。 | 新增行需要 `MaxStudentCount` 和 `StartStudentCount` 字段。 | **需要修改**。需要在 `addEmptyRow` 和 `handleAddTenRows` 的逻辑中为新行数据添加这两个字段的默认值（`null` 或 `''`），以确保 Vue 的响应式系统能正确追踪。 |
| **课程选择逻辑** | `ShiftColumn` 的 `onChange` 事件不涉及人数设置。 | `ShiftColumn` 的 `onChange` 事件需要根据课程类型联动修改人数。 | **需要扩展**。需要修改 `ShiftColumn` 的 `onChange` 回调，增加对课程 `isOneToOne` 属性的判断和对 `MaxStudentCount` / `StartStudentCount` 的赋值逻辑。 |
| **数据保存** | `collectAllChanges` 函数不处理 `MaxStudentCount` 等字段。 | `collectAllChanges` 需要包含 `MaxStudentCount` 等字段。 | **需要扩展**。需要在 `collectAllChanges` 中将这两个新字段加入到提交给后端的变更数据中。 |
| **API 接口** | `BatchSaveCourseDraft` 可能不支持新字段。 | 需要后端支持新字段的接收和保存。 | **依赖后端**。需要与后端同学确认 `BatchSaveCourseDraft` 接口已支持 `MaxStudentCount` 和 `StartStudentCount` 字段。 |

#### **3. 技术实现方案**

我们将采用动态组件和条件渲染的策略，在现有 `class-table-course.vue` 的基础上进行扩展，以最小化代码改动和维护成本。

**核心思路：**

1.  **动态列渲染**：改造 `getCurrentTableColumns` 函数，使其成为一个“列配置工厂”。当 `selectedTableType` 值为 `30` 时，它将返回一组专为“排预约课”设计的列配置，其中包含新增的“可约人数”和“开课人数”列，同时排除“班级”和“学员”列。
2.  **数据模型扩展**：在前端的数据模型（`tableData` 的行对象）中增加 `MaxStudentCount` 和 `StartStudentCount` 两个属性。修改 `useTableData.ts` 中的 `addEmptyRow` 等函数，确保新创建的行都包含这两个属性的默认值。
3.  **封装联动逻辑**：将“一对一”课程选择后自动填充人数并禁用的逻辑，封装在 `ShiftColumn` 列定义的 `onChange` 事件回调中。这样可以保证逻辑的内聚性。
4.  **行内校验**：将“开课人数不能大于可约人数”的校验逻辑放在 `StartStudentCountColumn` 列定义的 `onUpdateModelValue` 回调中，当用户修改“开课人数”时立即触发，提供实时的用户反馈。

#### **4. 修改点拆解与排序 (渐进式开发)**

**第一步：定义新增列的组件 (`class-table-course.vue`)**

1.  在 `<script setup>` 中，仿照 `ClassColumn`、`ShiftColumn` 等，定义两个新的列配置对象：
    *   `MaxStudentCountColumn`
    *   `StartStudentCountColumn`
2.  使用 `createEditableCell` 包裹 `ElInputNumber` 组件作为单元格的渲染器。
3.  在 `getProps` 回调中，设置 `:min="0"`、`step-strictly` 等属性，并根据行数据中的 `isOneToOne` 属性动态绑定 `:disabled`。
4.  在 `StartStudentCountColumn` 的 `onUpdateModelValue` 回调中，实现“开课人数不能大于可约人数”的即时校验和修正逻辑。
5.  在 `import` 区域从 `element-plus` 引入 `ElInputNumber`。

**第二步：改造列配置工厂 (`class-table-course.vue`)**

1.  找到 `getCurrentTableColumns` 函数。
2.  在 `else if (selectedTableType.value == 20)` 之后，添加一个新的 `else if (selectedTableType.value == 30)` 分支。
3.  在该分支中，返回一个新的列数组，其中应包含 `ShiftColumn`, `SubjectColumn`, `DateColumn` 等公共列，以及上一步创建的 `MaxStudentCountColumn` 和 `StartStudentCountColumn`。确保不包含 `ClassColumn` 和 `StudentUserColumn`。

**第三步：扩展数据模型与初始化 (`composables/useTableData.ts`)**

1.  打开 `src/pages/scheduleManage/tableCourseSchedule/composables/useTableData.ts` 文件。
2.  找到 `addEmptyRow` 函数。在返回的默认行对象中，增加 `MaxStudentCount: ''` 和 `StartStudentCount: ''` 两个字段。
3.  找到 `handleAddTenRows` 函数。在创建新行数据 `newRow` 的逻辑中，同样增加 `MaxStudentCount: ''` 和 `StartStudentCount: ''`。

**第四步：实现核心联动逻辑 (`class-table-course.vue`)**

1.  找到 `ShiftColumn` 的列定义。
2.  在 `getEvents` 回调返回的 `onChange` 函数中，增加以下逻辑：
    *   当 `courseData` 存在时，从 `courseData` 中获取 `IsOneToOne` 属性，并将其存入当前行 `row` 中（例如 `row.isOneToOne = courseData.IsOneToOne`）。
    *   判断 `courseData.IsOneToOne` 的值。如果为 `1`，则设置 `row.MaxStudentCount = 1` 和 `row.StartStudentCount = 1`。
    *   如果 `IsOneToOne` 不为 `1`，或者课程被清空 (`courseData` 为 `null`)，则将 `row.MaxStudentCount` 和 `row.StartStudentCount` 重置为空字符串 `''`，并将 `row.isOneToOne` 设为 `0` 或 `null`。

**第五步：更新数据提交流程 (`class-table-course.vue`)**

1.  找到 `collectAllChanges` 函数。
2.  在处理**新增行** (`if (hasNewRow)`) 和**修改行** (`else`) 的逻辑中，确保 `MaxStudentCount` 和 `StartStudentCount` 字段被正确地收集到 `rowData` 对象中，以便提交给后端。对于空值，应提交 `0` 或后端可接受的默认值。

#### **5. 前后端依赖**

*   **前端依赖后端**：`BatchSaveCourseDraft` 接口必须支持 `MaxStudentCount` 和 `StartStudentCount` 两个字段的接收和存储。**（需要后端确认）**

#### **6. 测试要点**

1.  **界面显示**：
    *   切换到“排预约课”，确认表格中显示“可约人数”和“开课人数”列，且不显示“上课班级/学员”列。
    *   切换回“给班级/学员排课”，确认“可约人数”和“开课人数”列消失，且“上课班级/学员”列正常显示。
2.  **联动逻辑**：
    *   在“排预约课”模式下，选择一个“一对一”课程，确认“可约人数”和“开课人数”自动变为 `1` 且输入框被禁用。
    *   更换为非“一对一”课程，确认这两个字段被清空且输入框恢复可用。
    *   清除课程选择，确认这两个字段被清空。
3.  **数据编辑与校验**：
    *   手动输入“可约人数”为 `5`，“开课人数”输入 `6`，确认有错误提示，并且数值被自动修正为 `5`。
    *   确认输入框只接受正整数。
4.  **数据保存与加载**：
    *   填写数据后，检查 `collectAllChanges` 函数生成的 payload 中是否正确包含了 `MaxStudentCount` 和 `StartStudentCount` 的值。
    *   （依赖后端）数据保存成功后，重新加载页面，确认这两个字段的值能够被正确回显。
5.  **分组功能**：
    *   在“给班级排课”模式下按“上课班级”分组，然后切换到“排预约课”，确认分组被自动清除并收到提示。**（回归测试）**

**第六步：更新数据校验模块 (`composables/useDataValidation.ts`)**

1.  打开 `src/pages/scheduleManage/tableCourseSchedule/composables/useDataValidation.ts` 文件。
2.  在 `fieldMap` 对象中，增加 `MaxStudentCount: 'MaxStudentCount'` 和 `StartStudentCount: 'StartStudentCount'`，保持映射表的结构完整。
3.  在 `fieldFormatCheck` 对象中，为 `MaxStudentCount` 和 `StartStudentCount` 添加校验函数。函数逻辑应为：校验值是否为数字或数字字符串，且大于等于0。
4.  在 `detectInvalidFields` 函数的 `idNameMapping` 中，暂时不需要添加新字段，因为它们没有对应的 Name 字段。

**第七步：更新表格工具函数 (`composables/useTableUtils.ts`)**

1.  打开 `src/pages/scheduleManage/tableCourseSchedule/composables/useTableUtils.ts` 文件。
2.  找到 `clearRelatedFields` 函数。
3.  在 `case 'ShiftID':` 的逻辑块中，增加代码来清空 `row.MaxStudentCount` 和 `row.StartStudentCount`。这是为了确保当用户清除“上课课程”时，由课程联动带出的“可约人数”和“开课人数”也能被一并清空。

**第八步：审阅必填字段校验逻辑 (`composables/useValidation.ts`)**

2.  仔细检查 `shouldValidateRowWithType` 函数。确认当 `tableType` 为 `30` 时，逻辑分支能正确处理，不会将 `MaxStudentCount` 或 `StartStudentCount` 视为判断行是否为空的关键字段。**（经初步分析，当前逻辑已能满足，此步骤为二次确认）**
3.  检查 `validateRequiredFields` 函数。确认当 `tableType` 为 `30` 时，`requiredFields` 数组的构建逻辑是正确的，不会错误地将新字段加入必填项列表。**（经初步分析，当前逻辑已能满足，此步骤为二次确认）**
