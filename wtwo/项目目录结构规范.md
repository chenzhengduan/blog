# WTwo 前端项目 `src` 目录结构规范

> **文档目的**: 定义 WTwo 前端项目 `src` 目录的标准结构与使用约定，旨在提高代码的可维护性、可读性和团队协作效率。
> **版本**: 1.2.0

---

## 1. 设计原则

本目录结构遵循以下核心设计原则：

- **关注点分离 (SoC)**：不同职责的代码（如视图、状态管理、API请求）应严格分离。
- **功能内聚**：强相关的文件（如某个页面及其子组件）应在物理上靠近。
- **约定优于配置**：通过明确的目录和文件命名约定，减少沟通成本。
- **可维护性与可扩展性**：结构设计为未来的功能扩展和代码重构提供便利。

---

## 2. `src` 核心目录结构

```
src/
├── api/                 # API 请求模块
├── assets/              # 静态资源
├── components/          # 全局通用组件
├── config/              # 应用级配置
├── hooks/               # 可复用的 Composition API
├── pages/               # 页面视图
├── router/              # 路由配置
├── services/            # 业务服务层
├── store/               # 全局状态管理
├── types/               # TypeScript 类型定义
└── utils/               # 全局工具函数
├── App.vue              # 根组件
└── main.ts              # 应用入口文件
```

---

## 3. 目录详解

### 3.1 `api/`
- **职责**: 定义所有与后端交互的接口请求函数。
- **规范**: 按业务模块拆分文件（例如 `user.ts`, `schedule.ts`）。此层仅负责数据请求与响应，不应包含业务逻辑。

### 3.2 `assets/`
- **职责**: 存放需要经过 Vite/Webpack 等构建工具处理的静态资源。
- **规范**: 内部按资源类型划分 子目录，如 `images/`, `styles/`, `fonts/`。

### 3.3 `components/`
- **职责**: 存放可在应用内跨多个页面复用的全局组件。
- **建议**:
    - **通用基础组件**: 可在 `components/common/` 中存放纯 UI、无业务逻辑的组件（如 `base-table`）。
    - **通用业务组件**: 可在 `components/business/` 中存放封装了特定业务、可复用的组件（如各类 `select` 组件）。

### 3.4 `hooks/`
- **职责**: 存放自定义的、可复用的 Vue Composition API 函数。
- **规范**: 文件名和函数名均以 `use` 开头（例如 `usePagination.ts`），用于抽离和复用组件中的有状态逻辑。

### 3.5 `pages/` (或 `views/`)
- **职责**: 存放应用的所有页面级组件。
- **规范**:
    - **页面组件内聚**: 对于结构复杂的页面（如 `scheduleManage`），应在其目录下创建专属的 `components/` 子目录，用于存放仅此页面使用的私有组件。

### 3.6 `router/`
- **职责**: 管理应用的所有路由配置。
- **规范**: 建议按业务模块在 `router/modules/` 目录下创建独立的路由配置文件，以保持路由系统的清晰和可维护性。

### 3.7 `services/`
- **职责**: 存放复杂的业务逻辑处理层。作为 `pages` (视图) 和 `api` (数据获取) 之间的桥梁，用于处理需要协调多方（如API、Store）的业务流程。

### 3.8 `store/`
- **职责**: 使用 Pinia 管理应用的全局状态。
- **规范**: 按业务模块（如 `user`, `dict`）拆分独立的 Store 文件。

### 3.9 `types/`
- **职责**: 存放全局共享的 TypeScript 类型定义，如 `interface`, `type`, `enum`。

### 3.10 `utils/`
- **职责**: 存放纯粹的、无副作用的、与业务无关的全局工具函数（例如：日期格式化、数据校验、深拷贝等）。
