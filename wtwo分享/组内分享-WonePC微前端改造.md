## 一、开场（3分钟）

- **WonePC 老项目背景**：十多年历史，Avalon 架构，技术栈陈旧，维护困难，开发效率低下。
- **为什么要用微前端？**
    - **对比其他方案**：我们也考虑过其他方案，比如跳转到新页面或者用iframe嵌入，但都有明显的问题：
        - **跳转方案**：用户体验割裂，需要重新加载页面，操作流程不连贯
        - **iframe方案**：虽然简单，但存在样式体验、通信复杂等问题
    - **渐进式迁移**：允许我们逐步替换旧的功能模块，而不是一次性重构整个应用，降低风险。
    - **技术栈无关**：主应用（WonePC/Avalon）可以与子应用（wtwo/Vue3）并行开发，互不影响。
    - **独立部署**：子应用可以独立开发、测试、部署，提升团队协作效率。
- **本次分享目标**：让大家了解整体架构升级方向、协作方式和当前进展。
- **分享初衷**：分享我们的探索过程和阶段性成果，也希望听到大家的意见和建议，共同完善这个方案。同时也希望通过这次分享，让其他前端同事能够更好地上手开发，降低学习成本。

## 二、微前端概览（5分钟）

### 2.1 微前端理念简介

微前端是一种将单体前端应用分解为多个可独立开发、部署和运行的前端应用的架构模式。它借鉴了微服务的理念，核心是**模块化、解耦、独立部署**。

- **独立开发**：团队可使用不同技术栈独立开发。
- **独立部署**：应用可独立构建、测试和部署。
- **独立运行**：应用在主框架中能够独立运行。

### 2.2 为什么选择 `micro-app`？

对比了市面上主流的微前端框架，如 `qiankun`、`single-spa`，我们最终选择了京东团队的 `micro-app`。

| 方案 | 实现原理 | 学习成本 | 侵入性 | 样式隔离 | 适用场景 |
|------|---------|---------|--------|----------|----------|
| **micro-app** | Web Components | 低 | 低 | 完善 | **我们的场景** |
| qiankun | single-spa | 中 | 中 | 一般 | 复杂应用 |
| single-spa | 路由劫持 | 高 | 高 | 需配置 | 大型项目 |
| Module Federation | Webpack5 | 中 | 中 | 需配置 | 现代项目 |
- Web Components 是一套技术，允许创建可重用的自定义元素
**`micro-app` 的核心优势:**
- 🎯 **使用简单**：接入方式类似 iframe，学习成本低。
- 🎯 **轻量级**：仅≈10kb (gzip)，零依赖。
- 🎯 **完善沙箱**：提供 JS 沙箱、样式隔离和元素隔离。
- 🎯 **技术兼容**：能很好地适配 Avalon 这样的老旧框架。
- 🎯 **功能丰富**：数据通信、预加载、生命周期等功能一应俱全。

### 2.3 主-子应用结构

我们的架构由一个**主应用（WonePC）**和多个**子应用（wtwo 是第一个）**组成。

```
WOnePC微前端架构
├── 主应用 (WonePC - Avalon)
│   ├── 框架基础设施
│   ├── 现有业务功能模块
│   ├── 用户认证 & 权限
│   ├── 全局状态管理
│   └── 子应用容器 (加载 wtwo)
└── 子应用 (wtwo - Vue3 + TypeScript)
    ├── 业务功能模块 (排课、日历等)
    ├── 内部状态管理 (Pinia)
    ├── UI 组件库 (Element Plus + 自定义组件)
    ├── 工具函数库 (filters、utils)
    └── 与主应用通信模块
```

### 2.4 技术栈详情

**子应用 wtwo 技术栈：**
- **框架**：Vue 3.3+ (Composition API)
- **构建工具**：Vite 4+
- **语言**：TypeScript 4.9+
- **状态管理**：Pinia
- **UI 库**：Element Plus (自定义 wtwo- 命名空间)
- **样式**：SCSS + UnoCSS
- **路由**：Vue Router 4 (Hash 模式 配合 micro-app 的 pure 模式)
- **HTTP 客户端**：基于 Fetch API 的自定义封装

## 三、旧架构与新架构的对接方式（10分钟）

### 3.1 主应用 WonePC (Avalon) 的改造

在主应用中，我们通过 UMD 方式引入 `micro-app` 库。

**1. 引入并启动 `micro-app`**
```html
<!-- 在主应用 HTML 中引入 -->
<script src="js/libs/micro-app/index.umd.js"></script>
```

```javascript
// 在主应用入口 JS 中启动
document.addEventListener('DOMContentLoaded', function() {
  if (window.microApp) {
    window.microApp.default.start({
      shadowDOM: false,         // 关闭shadowDOM，提高兼容性
      disableScopecss: false,   // 开启样式隔离
      destroy: true             // 卸载时销毁实例
    });
  }
});
```

**2. 嵌入子应用**

在需要展示新功能的页面（如排课页面），我们使用 `<micro-app>` 标签来嵌入 Vue 子应用。

```html
<!-- appointCourse.html -->
<micro-app 
  name="wtwo-app" 
  url="http://localhost:8001"  
  iframe>
</micro-app>
```
- `name`: 子应用的唯一标识。
- `url`: 子应用的访问地址，开发环境指向本地 dev server，生产环境指向部署后的地址。
- `iframe`: 开启 iframe 模式，可以实现最强的隔离，避免环境污染。
-  `其他`: 还有一些功能属性详见文档或代码

### 3.2 子应用 wtwo (Vue3) 的配置

子应用需要进行少量配置来支持在微前端环境中运行。

**1. 设置跨域**

在 `vite.config.ts` 中设置允许跨域，以便主应用能加载到子应用的资源。（本地开发阶段）

```typescript
// vite.config.ts
export default defineConfig({
  server: {
    cors: true, // 允许跨域
    headers: {
      'Access-Control-Allow-Origin': '*'
    }
  }
});
```

**2. 设置 public-path**
为了让子应用中的静态资源（如图片、字体）能正确加载，我们需要在入口文件动态设置 `public-path`。

```javascript
// src/main.ts
if (window.__MICRO_APP_ENVIRONMENT__) {
  __webpack_public_path__ = window.__MICRO_APP_PUBLIC_PATH__
}
```

### 3.3 主子通信与状态共享

我们封装了统一的通信机制和数据共享机制。

**1. 基础通信机制**

- **主应用 → 子应用**: 通过 `setData` 方法传递数据。
  ```javascript
  // 主应用：传递校区列表和用户信息
  window.microApp.setData('wtwo-app', {
    campusList: [...],
    userInfo: {...},
    SID: 'session_id_12345'
  });
  ```
- **子应用 ← 主应用**: 通过 `addDataListener` 监听数据。
  ```javascript
  // 子应用：在 onMounted 中监听
  window.microApp.addDataListener((data) => {
    console.log('接收到主应用数据:', data);
    // 更新 Pinia store
    useUserStore().setSID(data.SID);
    useOrganizationStore().setCampusList(data.campusList);
  });
  ```
- **子应用 → 主应用**: 通过 `dispatch` 发送事件。
  ```javascript
  // 子应用：通知主应用执行某个操作
  window.microApp.dispatch({ 
    type: 'updateBreadcrumb', 
    payload: '排课管理/日历视图' 
  });
  ```
- **主应用 ← 子应用**: 监听 `datachange` 事件。
  ```javascript
  // 主应用：监听 micro-app 元素
  const microAppElement = document.querySelector('micro-app[name="wtwo-app"]');
  microAppElement.addEventListener('datachange', (e) => {
    const { type, payload } = e.detail.data;
    if (type === 'updateBreadcrumb') {
      updateBreadcrumbPath(payload);
    }
  });
  ```

**2. 状态管理机制**

子应用使用 Pinia 进行状态管理，主要包括：

```typescript
// stores/index.ts - Pinia Store 结构
export const useUserStore = defineStore('user', {
  state: () => ({
    user: null as User | null,
    SID: '' as string
  })
});

export const useOrganizationStore = defineStore('organization', {
  state: () => ({
    campusList: [] as Campus[],
    currentCampus: null as Campus | null
  })
});

export const useScheduleStore = defineStore('schedule', {
  state: () => ({
    schedules: [] as Schedule[],
    currentView: 'calendar' as 'calendar' | 'table' | 'list'
  })
});
```

## 四、常见问题与解决方案（7分钟）

在对接过程中，我们遇到并解决了一些典型问题。

| 问题场景 | 原因分析 | 解决方案 |
|---|---|---|
| **样式冲突** | 主应用 WonePC 重写了 ElementUI 的样式，而子应用 wtwo 使用了新版的 Element Plus。默认的样式隔离会导致子应用的弹窗等组件样式错乱。 | **自定义命名空间**。我们为子应用的 Element Plus 组件设置了 `wtwo-` 的命名空间（如 `.wtwo-button`），并对 Element Plus 的基础样式（如 `el-overlay`）做了特殊处理，不进行隔离，保证其在主应用中能正常显示。 |
| **资源加载重复** | iframe 模式下，子应用默认会加载主应用的 `index.html`，导致不必要的资源重复加载。 | **指定空白 HTML**。我们为子应用指定一个只包含必要脚本的 `empty.html`，避免加载主应用多余的资源。 |
| **页面路由跳转** | 子应用 wtwo 有自己的路由（Vue Router），但主应用 Wone 没有。在嵌入页时子应用的路由出现在了主应用地址栏，导致面刷新时，URL 中的子应用路由会导致主应用解析失败。 | **使用 `micro-app` 的 pure 路由模式**。pure模式让子应用独立于浏览器路由系统进行渲染，不修改浏览器地址，也不增加路由堆栈，子应用更像是一个组件。这样既保持了子应用的路由功能，又不会影响主应用的URL。 |
| **开发环境代理** | 本地开发时，主应用需要能访问到子应用的服务。 | **配置 Proxy**。在 `vite.config.ts` 中配置代理，将对 `/wtwo` 的请求转发到子应用的开发服务器。线上环境则通过 Nginx 配置实现。 |
| **Element Plus 弹窗层级** | Element Plus 的弹窗组件（如 Dialog、Drawer）的 z-index 可能被主应用的某些元素覆盖。 | **调整层级**。通过 CSS 变量或者在弹窗组件上设置更高的 z-index，确保子应用弹窗显示在最顶层。 |
| **路由刷新问题** | 子应用内部路由刷新时，浏览器会尝试访问完整的路径，可能导致 404。 | **Hash 路由模式**。使用 `createWebHashHistory()` 模式避免路径问题，或者在服务器配置路由回退规则。 |
| **热更新失效** | 开发时子应用的热更新可能失效，需要手动刷新。 | **配置 HMR**。在 `vite.config.ts` 中配置 `server.hmr` 选项，确保热更新正常工作。 |

## 五、团队协作与部署方式（5分钟）

### 🧩 协作模式

- **独立仓库**：wtwo 子应用在独立的 Git 仓库中开发。
- **构建产物部署**：子应用独立构建，并将产物（dist 文件夹）部署到各个站点的服务器特定路径下（如 `/wtwo`）。
- **主应用引用**：主应用通过 `<micro-app>` 标签引用部署好的子应用产物地址。
- **统一规范**：
    - **接口协议**：与后端定义统一的接口规范。
    - **登录态**：登录由主应用统一处理，同域名天然支持子应用调用接口，获取token调用新后端接口。
    - **事件通信**：定义统一的事件名和数据结构。

### 🚀 部署方式

- **子应用独立部署**：wtwo 项目通过 CI/CD 构建后，静态资源被推送到服务器的 `/wtwo` 目录下。
- **主应用无感知更新**：主应用只需配置好 `url` 指向生产环境的 `/wtwo` 路径即可。子应用的更新和部署对主应用是透明的。
- **联调**：本地开发时，通过配置 `vite.config.ts` 的 `server.proxy` 实现跨域联调。

### 6.1 日历视图排课 (`AppointCourse.vue`)

**当时的问题是什么？**
需要一个直观、可交互的日历界面来安排课程，但遇到了一个经典问题：**时间冲突**。比如周二上午9:00-11:00有Java编程课，9:30-10:30又有数据库设计课，这两门课时间重叠了，该怎么显示？

**我做了哪些设计？**

1. **分列显示方案**：
   - **核心思路**：把每天的区域想象成可以分成多列的表格
   - **冲突解决**：时间重叠的课程放在不同的列中并排显示
   - **自动计算**：系统会自动算出需要几列，以及每门课应该放在哪一列

   ```
   原始重叠显示：           分列显示：
   ┌─────────────┐         ┌──────┬──────┬─────────┐
   │ Java编程    │         │Java  │数据库│         │
   │ ┌─────────┐ │   →    │编程  │设计  │         │
   │ │数据库设计││         │      │      │         │
   │ └─────────┘ │         └──────┴──────┴─────────┘
   └─────────────┘
   ```

2. **智能布局算法**：
   - **第一步：找冲突** - 扫描所有课程，找出哪些课程时间重叠
     ```
     重叠判断公式：A的开始时间 < B的结束时间 且 A的结束时间 > B的开始时间
     例如：Java编程(9:00-11:00) vs 数据库设计(9:30-10:30)
     判断：9:00 < 10:30 ✓ 且 11:00 > 9:30 ✓ → 有重叠
     ```
   - **第二步：算列数** - 找出同时进行的最多课程数，决定需要几列
   - **第三步：分位置** - 按开始时间顺序，为每门课分配一个可用的列
   - **第四步：算样式** - 根据列位置计算每门课的宽度和位置，绝对定位

3. **交互体验优化**：
   - **拖拽功能**：可以拖拽课程块调整时间，系统会实时显示预览
   - **时间对齐**：拖拽时会自动对齐到时间网格，避免出现奇怪的时间点
   - **冲突提示**：拖拽到冲突位置时会给出视觉反馈

**带来了哪些好处？**
- **直观显示**：时间冲突一目了然，不会出现课程重叠看不清的情况
- **自动布局**：不需要手动调整，系统自动处理复杂的布局逻辑
- **灵活交互**：支持拖拽调整，操作简单直观
- **性能良好**：即使有很多课程，布局计算也很快，但是渲染可能不一定

### 6.2 表格排课 (`tableCourseSchedule.vue`)


日历视图适合直观地查看和调整单门课程，但对于有规律的批量排课（比如每周重复的课程）就显得效率很低。需要一个类似 Excel 的表格界面来支持快速、批量的排课操作。


1. **组件选型探索**：
   - **调研过程**：当时找了很多开源表格组件，基于vtable开发了核心功能demo，但体验不够好
   - **最终选择**：使用 vue-easytable 组件
   - **原因**：vue-easytable 功能强大，支持编辑、复制、下拉填充等复杂功能，它本身只支持vue2,经过处理 Vue3 可以用了。

2. **Excel 风格的表格界面**：
   - **可编辑单元格**：双击任意单元格就能进入编辑模式
   - **智能选择器**：根据字段类型（班级、教师、教室等）弹出对应的选择器
   - **实时预览**：选择后立即在表格中显示，所见即所得

2. **智能数据管理**：
   - **双重存储**：举例：选择班级时，既保存班级ID（用于数据提交），也保存班级名称（用于显示）
   - **避免重复查询**：表格显示时直接使用保存的名称，不需要每次都去查询

3. **批量操作功能**：
   - **拖拽填充**：类似Excel，选中一行后向下拖拽，自动复制到下面的行
   - **智能复制**：不仅复制ID，还会复制对应的名称，保证数据的完整性
   - **批量修改**：支持选中多行进行批量操作

**带来了哪些好处？**
- **效率提升**：批量排课的速度比日历视图快3-5倍
- **操作简单**：熟悉Excel的用户能立即上手
- **数据准确**：智能的数据管理避免了显示错误
- **性能优化**：减少了大量的重复查询，表格渲染更流畅

- Q: 子应用如何处理用户权限？

- Q: 如果子应用出现故障，会影响主应用吗？
- A: 不会。iframe 模式提供了强隔离，子应用的 JS 错误不会影响主应用。如果子应用无法加载，可以显示老页面。
