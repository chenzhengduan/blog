一、开场

为什么我们要做微前端改造？

🚀 WonePC 老项目背景

十多年历史，Avalon 架构，技术栈陈旧，维护困难，开发效率低下。

🎯 分享初衷

  • 分享探索过程：展示我们的阶段性成果，希望听到大家的意见和建议
  • 共同完善方案：通过团队讨论，持续优化微前端解决方案
  • 降低学习成本：让其他前端同事能够更好地上手开发

💡 为什么要用微前端？

对比其他方案

  • 跳转方案：用户体验割裂，需要重新加载页面
  • iframe方案：虽然简单，但存在样式体验、通信复杂等问题

微前端优势

  • 渐进式迁移：逐步替换旧功能模块，降低风险
  • 技术栈无关：主应用与子应用并行开发，互不影响
  • 独立部署：提升团队协作效率

二、微前端概览

我们为什么选择了micro-app？

🎯 微前端理念简介

微前端是一种将单体前端应用分解为多个可独立开发、部署和运行的前端应用的架构模式。

核心理念：模块化、解耦、独立部署

⚡ 为什么选择 micro-app？

方案 实现原理 学习成本 侵入性 样式隔离 适用场景

single-spa 路由劫持 高 高 需配置 大型项目

🧩 微前端嵌入方式

组件化嵌入

微前端子应用就像一个个独立组件，可以无缝嵌入到主应用中。用户在使用时完全感觉不到这是两个不同的系统。

<!-- 就像使用普通组件一样 --> <micro-app name="wtwo-app" url="/wtwo"></micro-app>

用户体验优势

  • 无感知切换：用户感觉在使用同一个系统
  • 统一交互：保持一致的界面风格和操作习惯
  • 流畅体验：无需重新加载页面，响应更快
  • 数据共享：主应用和子应用可以共享用户状态

三、主应用 WonePC (Avalon) 的改造

如何让老系统支持微前端？

🔧 引入并启动 micro-app

<script src="js/libs/micro-app/index.umd.js"></script> window.microApp.default.start({ shadowDOM: false, // 关闭shadowDOM disableScopecss: false, // 开启样式隔离 destroy: true // 卸载时销毁实例 });

📦 嵌入子应用

在需要展示新功能的页面使用 <micro-app> 标签嵌入 Vue 子应用。

<micro-app name="wtwo-app" url="http://localhost:8001" iframe> </micro-app>

标签属性说明

  • name：子应用的唯一标识
  • url：子应用的访问地址
  • iframe：开启 iframe 模式，实现强隔离
  • 其他：更多功能属性详见文档

主子通信与状态共享

如何实现应用间的数据传递？

📡 基础通信机制

主应用发送 → 子应用接收

主应用向子应用传递数据（用户信息、校区列表等）

// 主应用发送数据 window.microApp.setData('wtwo-app', { campusList: [...], userInfo: {...}, SID: 'session_id_12345' });

子应用接收 ← 主应用发送

子应用监听并接收主应用传递的数据

// 子应用接收数据 window.microApp.addDataListener((data) => { useUserStore().setSID(data.SID); useOrganizationStore().setCampusList(data.campusList); });

子应用发送 → 主应用接收

子应用向主应用发送事件（更新导航、触发操作等）

// 子应用发送事件 window.microApp.dispatch({ type: 'updateBreadcrumb', payload: '排课管理/日历视图' });

主应用接收 ← 子应用发送

主应用监听并处理子应用发送的事件

// 主应用接收事件 const microAppElement = document.querySelector('micro-app[name="wtwo-app"]'); microAppElement.addEventListener('datachange', (e) => { const { type, payload } = e.detail.data; if (type === 'updateBreadcrumb') { updateBreadcrumbPath(payload); } });

四、常见问题与解决方案

我们踩过的几个坑和解决方案

🎨 样式冲突

问题：主应用 WonePC 重写了 ElementUI 的样式，而子应用 wtwo 使用了新版的 Element Plus。默认的样式隔离会导致子应用的弹窗等组件样式错乱。

解决：自定义命名空间。我们为子应用的 Element Plus 组件设置了 wtwo- 的命名空间（如 .wtwo-button），并对 Element Plus 的基础样式（如 el-overlay）做了特殊处理，不进行隔离，保证其在主应用中能正常显示。

📦 资源加载重复

问题：iframe 模式下，子应用默认会加载主应用的 index.html，导致不必要的资源重复加载。

解决：指定空白 HTML。我们为子应用指定一个只包含必要脚本的 empty.html，避免加载主应用多余的资源。

🔄 页面路由跳转

问题：子应用 wtwo 有自己的路由（Vue Router），但主应用 Wone 没有。在嵌入页时子应用的路由出现在了主应用地址栏，导致面刷新时，URL 中的子应用路由会导致主应用解析失败。

解决：使用 micro-app 的 pure 路由模式。pure模式让子应用独立于浏览器路由系统进行渲染，不修改浏览器地址，也不增加路由堆栈，子应用更像是一个组件。这样既保持了子应用的路由功能，又不会影响主应用的URL。

🔧 开发环境代理

问题：本地开发时，主应用需要能访问到子应用的服务。

解决：配置 Proxy。在 vite.config.ts 中配置代理，将对 /wtwo 的请求转发到子应用的开发服务器。线上环境部署到同一个域名下，接口请求天然支持，无需额外配置。

📱 Element Plus 弹窗层级

问题：Element Plus 的弹窗组件（如 Dialog、Drawer）的 z-index 可能被主应用的某些元素覆盖。

解决：调整层级。通过 CSS 变量或者在弹窗组件上设置更高的 z-index，确保子应用弹窗显示在最顶层。

五、团队协作与部署方式

我们是怎么一起协作的？

🧩 协作与部署模式

📁 独立仓库开发

wtwo 子应用在独立的 Git 仓库中开发，与主应用完全解耦，便于团队并行开发和版本管理。

🏗️ 独立构建部署

子应用独立构建，通过 CI/CD 将产物部署到服务器的 /wtwo 目录下。

🔗 主应用引用

主应用通过 <micro-app> 标签引用部署好的子应用，实现无感知更新。

🔧 开发联调

本地开发时通过 vite.config.ts 的 server.proxy 实现跨域联调，线上环境同域名天然支持。

🌐 环境切换

开发、测试、生产环境通过不同的 url 配置实现无缝切换，无需修改主应用代码。

📋 统一规范

  • 接口协议：与后端定义统一的接口规范
  • 登录态：登录由主应用统一处理，同域名天然支持
  • 事件通信：定义统一的事件名和数据结构

📈 协作流程

  • 需求评审：新功能开发前，前后端测试一起评审技术方案
  • 接口设计：统一接口规范，避免新老接口差异过大
  • 联调测试：重点关注跨应用交互和通信场景
  • 上线部署：子应用独立部署，主应用无感知更新

智能布局算法

四步解决复杂布局问题

🔍 第一步：找冲突

重叠判断公式： A的开始时间 < B的结束时间 且 A的结束时间 > B的开始时间 例如：Java编程(9:00-11:00) vs 数据库设计(9:30-10:30) 判断：9:00 < 10:30 ✓ 且 11:00 > 9:30 ✓ → 有重叠

📊 第二步：算列数

找出同时进行的最多课程数，决定需要几列

📍 第三步：分位置

按开始时间顺序，为每门课分配一个可用的列

🎨 第四步：算样式

根据列位置计算每门课的宽度和位置，绝对定位

八、Q&A 环节

大家有什么问题都可以问

🔧 技术问题

Q: 子应用如何处理用户权限？

A: 权限由主应用统一管理，通过通信机制传递给子应用。子应用根据权限控制页面和功能的显示。

Q: 如果子应用出现故障，会影响主应用吗？

A: 不会。iframe 模式提供了强隔离，子应用的 JS 错误不会影响主应用。

📈 业务问题

Q: 新功能开发周期如何？

A: 由于技术栈现代化和组件复用，开发效率比原来项目开发提升约 30%。

Q: 用户是否感知到微前端的存在？

A: 理想情况下用户无感知。我们通过样式统一、交互一致性保证用户体验的连贯性。
