# 我为 WTwo 前端项目定义的 `src` 目录规范

大家好，我是你们的 AI 技术伙伴。

为了确保我们的 WTwo 项目能够长期保持高效、清晰和可维护，我为大家精心设计了这份 `src` 目录结构规范。这不仅仅是一份规则文档，更是我们团队协作的蓝图和代码组织的哲学。遵循这份规范，将帮助我们构建出更健壮、更易于扩展的应用。

---

## 1. 我的设计哲学

在我设计这套结构时，我主要遵循了以下几个核心原则：

- **关注点分离 (Separation of Concerns)**：我坚持让每一部分代码只做一件事。视图（Components）、业务逻辑（Services）、状态（Store）、接口（API）各司其职，互不干扰。
- **功能内聚 (Functional Cohesion)**：我认为强相关的代码应该在物理上彼此靠近。例如，一个复杂页面和它专属的子组件应该放在同一个目录下。
- **约定优于配置 (Convention over Configuration)**：我相信，清晰的目录和文件命名约定能让我们减少不必要的配置和沟通，让新成员也能快速上手。
- **着眼未来 (Maintainability & Scalability)**：我设计的结构充分考虑了未来的需求变更和功能扩展，让代码库能健康地成长。

---

## 2. `src` 目录的核心蓝图

这是我为我们项目 `src` 目录绘制的核心蓝图。它清晰地展示了我们代码库的骨架，并标注了每个部分的状态：

```
src/
├── api/                 # ✅ API 请求模块
├── assets/              # ✅ 静态资源
├── components/          # 🟡 全局通用组件 (待重构)
├── config/              # ✅ 应用级配置
├── hooks/               # 🔵 (建议新增) 可复用的 Composition API
├── pages/               # ✅ 页面视图
├── router/              # ✅ 路由配置
├── services/            # ✅ 业务服务层
├── store/               # ✅ 全局状态管理
├── types/               # ✅ TypeScript 类型定义
└── utils/               # ✅ 全局工具函数
├── App.vue              # ✅ 根组件
└── main.ts              # ✅ 应用入口文件

(过渡期结构)
└── main/                # 🟠 旧的核心目录 (待移除)
```
下面，我将带大家逐一了解每个目录的职责和我的设计考量。

---

## 3. 各核心目录详解

### 3.1 `api/` - 数据通讯的专业信使
在这里，我定义了所有与后端服务进行通信的函数。我将它们按照业务领域（如 `user.ts`, `schedule.ts`）进行组织，确保每个文件都聚焦于一个模块。请记住，`api` 层的职责非常纯粹：**只负责数据的请求与响应，不应包含任何业务处理逻辑**。

### 3.2 `assets/` - 项目的视觉资源库
这个目录是我存放项目静态资源的地方，比如样式文件（SCSS）、图片、字体等。所有需要经过 Vite 构建流程处理的资源都应放在这里，并按类型分门别类。

### 3.3 `components/` - 全局组件的家园 (及未来规划)
这里是存放全局通用组件的地方。然而，为了让组件库更加清晰，我规划了下一步的优化路径：
1.  **`common/` (通用基础组件)**：我会将像 `base-table` 这样不含任何业务逻辑、纯粹的 UI 构建块，迁移到一个名为 `common` 的子目录中。
2.  **`business/` (业务通用组件)**：像 `select/` 目录下的各种选择器，它们封装了特定的业务逻辑并能在多个页面复用，我会将它们归类到 `business` 子目录。
3.  **页面私有组件**：对于那些只在特定页面使用的组件（比如 `popup/` 里的），我认为它们应该被移动到所属页面的 `components` 子目录中，实现真正的“功能内聚”。

### 3.4 `hooks/` - 封装逻辑的魔术袋 (建议采纳)
我强烈建议我们开始使用 `hooks` 目录。当我发现多个组件中存在重复的、有状态的逻辑时（例如，处理分页的逻辑），我就会将其抽离成一个自定义的 Composition API 函数（以 `use` 开头，如 `usePagination.ts`），并放在这里。这能极大地提升代码的复用性和整洁度。

### 3.5 `pages/` - 应用的舞台
这里是用户能看到的所有页面的集合。我遵循的核心原则是**页面组件内聚**。对于像 `scheduleManage` 这样复杂的页面，我会在其目录下创建一个专属的 `components/` 子目录，存放所有只为它服务的子组件。这能让页面的结构一目了然。

### 3.6 `router/` - 应用的导航系统
我在这里配置了整个应用的页面路由。我们已经实现了按模块拆分路由，这是一种非常好的实践。未来所有新的页面路由，都应该在 `router/modules/` 下以独立文件的形式添加，以保持路由系统的清晰和可扩展性。

### 3.7 `services/` - 复杂业务的处理器
当一项业务逻辑变得复杂，需要协调 `api` 调用和 `store` 状态更新时，我会将这部分逻辑抽离到 `services` 层。它就像我们应用的大脑，负责处理复杂的业务流程，让 `pages` 层的组件保持轻量和聚焦于视图展示。`oss.ts` 就是一个很好的例子。

### 3.8 `store/` - 全局状态的中央银行
我使用 Pinia 来管理我们应用的全局共享状态。在这里，我按照业务领域（如 `user`、`dict`）来划分不同的 `store` 模块，确保每部分状态都有其清晰的管理者。

### 3.9 `types/` - 项目的“类型字典”
为了保证我们 TypeScript 项目的健壮性，我将所有全局共享的类型、接口和枚举都统一存放在 `types/` 目录中。这就像我们项目的共享字典，确保了数据结构在整个应用中的一致性。

### 3.10 `utils/` - 通用工具的兵器库
这里存放的是那些纯粹的、无副作用的、与具体业务无关的辅助函数，比如日期格式化、数据校验等。随着 `utils` 的完善，之前遗留的 `tool` 目录就可以被安全地移除了。

---

## 4. 关于 `src/main/` 的过渡期说明

我注意到项目中还存在 `src/main/` 这个历史目录。在我们的重构过渡期间，我将它视为一个临时兼容层。我为大家设定了两条简单的规则：
- **只修正，不新增**：我们可以在里面修复 Bug，但任何新功能或新代码都不应该再写进这个目录。
- **最终目标**：当团队所有成员都将依赖于它的代码迁移出来后，我们就将它**彻底删除**，完成我们项目结构的进化。
